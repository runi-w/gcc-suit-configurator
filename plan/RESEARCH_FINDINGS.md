# Configurator Drape & Pattern-Wrap Research — Findings

_Research only. No code touched, no constants tuned. Compiled 2026-07-21._

---

## 1. Bottom line

**Route by difficulty; don't bake everything, and don't trust "AI-bake solves wrap for free."** The first version of this report claimed image-to-image AI swap fixes per-panel wrap "for free" because the model renders the whole garment holistically. That claim does not survive contact with this project's own catalog history (§4 Challenge, corrected below): for solids, mélange, birdseye, herringbone and other non-directional cloths, the claim holds — the AI has nothing directional to get wrong, and the swap is cheap, fast, and has already shipped correctly across the bulk of 166 catalog fabrics. But for the fabrics that actually have a wrap problem — pinstripes, chalk stripes, windowpanes, glen/Prince-of-Wales checks — the catalog's own logged record shows the opposite of "for free": a documented 15-fabric Elite rework list plus roughly a dozen more patterned Prestige fabrics needed the user to photograph cloth with a ruler and a SpyderCheckr colour reference, a two-engine iterative pipeline (a reasoning model writing prompts for a second image model), and per-item colour-lock QA — and even after all that, some outputs (DBS139A; the first DBQ791A attempt, which the user called "TERRIBLE") were still unsatisfactory. AI-bake is easy exactly where the compositor is already fine, and expensive exactly where the compositor is broken.

The corrected recommendation is **difficulty-routing**: keep the runtime compositor — which this project's own audit already measures at Suitsupply parity for colour, pattern scale, sharpness, moiré, specular, and on-screen density — as the default renderer for the ~85% of the catalog (solids, subtle textures, non-directional weaves) where it already works, and reserve the expensive, human-in-the-loop AI-bake process for only the ~15–30 bold directional-pattern fabrics where wrap visibly breaks and where the catalog record shows the fix is worth its cost. Baking every fabric (the original recommendation) is kept in this report as a ranked alternative, not the primary path, because it spends real engineering and human hours re-solving a problem that's already solved for most of the catalog, for a result that — per HANDOVER's own measurements — wouldn't look meaningfully different for those fabrics anyway. Crease depth (drape) is a separate, universal gap that applies regardless of routing and needs a base-render/prompt fix, not a compositing fix (HANDOVER §6 item 2). Both bake-based paths also depend on a currently-incomplete prerequisite: only one of the configurator's 5 house-model cuts has the new, consistent model; the other four are pending regeneration (§4), and the catalog's proven AI-swap recipe has only ever been validated on the catalog's *walking* model, never on the configurator's *standing* one — an untested assumption, not a proven fact, and the first thing to check before committing effort either way.

### On the "at least one true-3D platform" sub-question (Threekit / Zakeke / Emersya)

- **Threekit**: this session's *prior* research (same conversation thread, not re-derived here) read Threekit's own technical documentation directly and found: full PBR material authoring (including anisotropy and sheen — the exact properties that matter for cloth) as the **source of truth**, but **delivery to the end customer is pre-rendered layers composited in 2D**, not live WebGL, for the fidelity tier that matters here. A follow-up background-agent pass this session re-read Threekit's docs and reported a more hedged "hybrid 3D Player / 2D Player, ratio not disclosed" — the two aren't contradictory (Threekit ships both a 3D player and a "virtual photography" 2D delivery mode; which a given implementation uses isn't publicly documented per-client), but the concrete, hard-docs-sourced finding is the more load-bearing one: **3D/PBR as source, 2D as delivery, is Threekit's own documented north star**, not an inference from marketing.
- **Zakeke / Emersya**: a background research agent characterized both as "real-time WebGL PBR, high confidence, no baked assets" — but this was sourced from vendor marketing/docs pages, **not from inspecting a live client store's actual network payload** the way Suitsupply/Hockerty/Proper Cloth/Black Lapel/Oliver Wicks were. Treat this as **unverified-to-live-asset, vendor-documented only** — a materially weaker evidence tier than the rest of this table. It would not be surprising if a real Zakeke/Emersya apparel deployment also falls back to baked hero imagery for the fabric-browsing step, the way Suitsupply and Threekit both do — but that specific claim was not checked against a live implementation this session.
- **Correction of an in-session error**: an earlier background-research pass, working without access to this session's live-asset findings, reported Suitsupply's rendering method as "unknown, not published, likely real-time 3D (presumed)." That is **wrong** and is superseded by the live evidence in the table above — Suitsupply is provably a baked-2D-photo-plus-layered-construction-details system, not real-time 3D. This is flagged explicitly so a future reader doesn't inherit the wrong claim from a lower-quality pass.

If that test doesn't show a clean, correctly-wrapped result on the standing pose, the whole premise of baking onto the configurator's own renders needs re-examining, independent of which routing strategy is chosen.

## 4. Ranked recommendations

### First, the question this section's own logic raises: what is the configurator *for*, if fabric+options become baked stills?

If every fabric (or even most of them) is reduced to a small set of pre-baked photographic stills, the visual layer of the configurator converges toward what the business's existing Shopify catalog *already is*: a front/back/swatch still per fabric, 117 Elite + a growing Prestige set, already live. That's worth confronting directly rather than sliding past: under a bake-heavy approach, the configurator's remaining differentiator is **not** live photoreal rendering — it's (a) letting a customer move through cut, lapel, button, and fabric choices in one continuous guided session instead of browsing dozens of separate static product pages, and (b) driving that session into **real order capture** — measurements, an add-to-cart payload with `_config` properties, the `orders/paid` webhook, and ultimately a genuine multi-garment KuteTailor `saveOrder` submission — none of which the static catalog does at all (those are fixed single-garment products with no measurement flow and no multi-garment spec assembly). That is a legitimate product on its own. It should just be named honestly: under any bake-heavy recommendation below, the configurator's job is "a guided picker over a matched still-image library, wired to real order capture," not "a live-render engine." Difficulty-routing (Recommendation 1) keeps this concern small, because most fabrics *keep* full live interactivity; full bake-everything (Recommendation 2) makes the concern larger, because more of the product surface becomes static stills — one more reason Recommendation 1 is ranked first.

### 1. Difficulty-route: keep the compositor for the easy majority, AI-bake only the ~15–30 bold directional patterns where wrap actually breaks (primary recommendation)

- **Why this ranks above bake-everything**: HANDOVER's own measured audit already shows the runtime compositor at Suitsupply parity on colour (98% vs 105%), pattern scale (0.99× true), pattern sharpness (95% of true cloth), moiré (none), specular (3.08 vs 3.30), and on-screen density (9.95 vs 10.3 px/cm) — for the majority of the catalog, there is no visible problem to spend money fixing. The two measured gaps — crease depth and pattern wrap — are not evenly distributed: crease depth is universal (applies to every fabric, and is a base-render/prompt problem, not a compositing one — see below), but pattern *wrap* is only visible where a directional pattern exists to misalign in the first place. A solid, a birdseye, a herringbone, a mélange has no grain direction to get wrong. Spending the bake effort on those fabrics buys nothing measurable.
- **Fixes drape**: only where baked (the hard subset) — and even there, drape is really a separate, universal fix (see "Depth/crease is not this recommendation's job" below). Routing does not, by itself, close the crease-depth gap for the fabrics left on the compositor.
- **Fixes wrap**: yes, precisely where it's broken, using the same catalog-proven fabric-swap technique — but **only after the required validation gate above passes**, and with the realistic cost/effort below, not the "for free" claim this report originally made.
- **Which fabrics get routed to bake**: reuse the classification the catalog work already produced — the 15-fabric Elite "bold-check/windowpane REWORK LIST" (`DBT3066, DBV195A, DBV196A, DBV197A, DBU037A, DBU038A, DBS136A, DBS137A, DBS138A, DBS139A, DBN314A, DEE1017`, plus the 3 houndstooths already reworked) and the roughly dozen patterned Prestige fabrics tracked separately (windowpanes, checks/Prince-of-Wales/glen, stripes). That's ~27–29 fabrics out of 166 (~16–17%) by this project's own prior classification — in the ballpark the compositor's audit itself implies (wrap is a measured problem, but it's a problem that only shows up where there's a pattern to show it).
- **Exact AI tools**: for the routed subset, the same tooling the catalog already built and iterated on: ruler/SpyderCheckr-measured scale where available, the two-engine pipeline (a reasoning model — `gpt-5.5-pro`-class — writing a bespoke prompt per fabric, fed to `gpt-image-2` or `gemini-3-pro-image` at the 4K output size that was found to render fine patterns natively), plus the colour-lock QA step the catalog pipeline already has scripted.
- **Realistic effort (see the full cost model below)**: this is *not* a rounding-error API cost. Based on this project's own logged history for exactly this fabric class, budget **1–4 hours of engineer/user time per hard fabric** (ruler photography, iterative prompt/generation rounds, colour-lock QA) across the roughly 27–29 fabrics that need it, and expect some outputs (per DBS139A, per the first DBQ791A attempt) to need a second pass or to land short of "great." The routed subset is genuinely a hand-crafted process, not a batch job.
- **Hard dependency, must happen first**: per HANDOVER §2/§6, only the `notch` cut currently has the regenerated, consistent house model; the other four cuts (peak, double-breasted, three-button, etc.) still use the old renders, so *the man changes when the cut changes*. Baking AI hero images on top of that inconsistency would permanently fix a cross-cut mismatch into expensively-produced "final" assets — worse than today, where the mismatch is at least a known, centrally-fixable gap in the live compositor. The 14-render regeneration (`plan/MODEL_BIBLE.md`) needs to land *before* any per-cut baking, routed or not.
- **Quality ceiling vs. Suitsupply**: at parity for the routed subset, assuming the validation gate passes; unchanged (already-at-parity) for everything else, at zero additional cost.
- **First implementation step**: (1) run the required validation gate above; (2) if it passes, re-tag the ~27–29 known hard fabrics (or re-derive the list against the live 166-fabric set, since the tracker was built against an earlier catalog state) and treat that list — not the whole catalog — as the bake backlog.

### 2. Full AI-bake per fabric × per cut — bake everything (alternative, ranked below Recommendation 1)

- **What changed from the original version of this recommendation**: the original claimed AI-bake fixes wrap "for free" across the board and estimated cost as API calls only (~830 renders, $40–125 total). Both are corrected here.
- **Fixes drape**: for the fabrics where the base render's own crease depth is the limiting factor, baking doesn't help more than routing does — it's the same base-render/prompt problem either way (HANDOVER §6 item 2), not something either compositing or baking fixes on its own.
- **Fixes wrap**: yes for the ~83–84% of fabrics with no directional pattern (same as routing achieves for free), and yes-but-expensive for the ~16–17% with one (same real cost as under routing) — baking the easy majority buys no *additional* wrap fix beyond what the compositor already delivers, since there's nothing there to fix.
- **Realistic, revised cost model** (replaces the original "$40–125, a rounding error" estimate):
  - *Easy majority (~136–139 fabrics)*: API cost only, ~$0.05–0.15/render × 5 cuts ≈ **$34–$104 total** — this part of the original estimate was directionally fine. But it isn't free of labor: the catalog's own QA record shows recolour fixes were needed on **36/116 (31%) of Elite Wool** and **18/37 (49%) of the generated Prestige fabrics**, and that's for just *two* views (front vs. back) of a *single* pose. A configurator baking **five different cuts** has to hold colour consistent across up to 10 pairwise comparisons per fabric, not one — call it **20–35 hours of QA/engineering time** for the easy majority alone, to catch and fix cross-cut colour drift, for a visual result that, per HANDOVER's own measurements, isn't meaningfully better than what the compositor already shows for these fabrics.
  - *Hard minority (~27–29 fabrics)*: the same 1–4 hours/fabric of ruler photography, two-engine iteration, and colour-lock QA as under Recommendation 1 — call it **30–115 hours total** — plus the pricier per-image cost of the two-engine pipeline (this project logged ~$0.19–0.28/image for `gpt-image-1` high-quality edits; the modern `gpt-image-2`/4K-Gemini combo is in the same range) across 5 cuts: roughly **$25–40 in API cost**, dwarfed by the human time either way.
  - **Total for bake-everything: roughly 50–150 engineering/human hours + ~$60–145 in API cost**, of which the ~20–35 hours spent on the easy majority buys little the compositor doesn't already provide, and the rest duplicates exactly the cost Recommendation 1 also has to pay for the hard subset.
- **Hard dependency**: same as Recommendation 1 — the 14-render regeneration must land first, and the validation gate on the standing house model must pass, or this recommendation inherits the same unproven-transfer risk at 5–6× the fabric count.
- **When this is the right call anyway**: if the business decides, for reasons outside this report's scope (e.g. wanting to remove the runtime compositor entirely as an engineering simplification, not for visual-quality reasons), to move to an all-static architecture. On visual-quality-per-dollar grounds alone, Recommendation 1 dominates.
- **First implementation step**: same validation gate as Recommendation 1; if pursued, budget the full 50–150 hour range above rather than the original $40–125 API-only figure.

### 3. Hero-bake caching: when a fabric is baked (under either 1 or 2), serve it as the "considered view," not a live composite

- This is a mechanism, not a competing strategy — it answers *when* to show a bake, while Recommendation 1 answers *which fabrics* deserve one.
- On first view of a baked fabric+cut combination, show the pre-generated hero image rather than regenerating on demand; since the combination space under routing is small (~27–29 fabrics × 5 cuts ≈ 135–145 combos, not 830), every combination can be pre-generated once and cached, with zero live API latency at checkout time.
- Mirrors what Suitsupply itself does: its layered runtime compositor handles construction-detail options, but the fabric-browsing step shows an AI-generated hero photo, not a live composite.
- **Build effort**: low — a UI decision (when to swap from live compositor to cached hero image), not new rendering technology.
- **First implementation step**: decide the trigger (e.g., landing on the Style/Summary tab, or selecting one of the routed hard-pattern fabrics specifically) and wire it to the bake cache.

### 4. Per-panel segmentation + independent warp fields (fallback for the hard subset, if bespoke AI-bake labor is judged too costly)

- **Fixes drape**: no — drape is a separate, already-identified base-render gap (HANDOVER §6 item 2), not something warp-field math can fix.
- **Fixes wrap**: partially — directly addresses the known, specific bug (the sleeve gets sheared as if it were part of the torso plane because `hw[y]` spans the whole row). Segmenting the garment into torso + each sleeve + each lapel with an off-the-shelf model (SAM2, or a human-parsing model like SCHP/Graphonomy) and applying the *already-validated* per-torso warp logic independently to each panel should recover something closer to the target (swing 22.7°, spread 16.7°, monotonic) without the sleeve-inversion/wobble regression noted in HANDOVER §6.
- **Exact AI tools**: SAM2 (zero-shot, no training) or a human-parsing model, run once per cut (not per fabric — fabric-independent, a real advantage over any bake-based approach).
- **Build effort**: medium, ~4–6 weeks for one engineer.
- **Quality ceiling**: below Recommendation 1/2's bake quality for the hard subset — still a 2D approximation of per-panel grain, not a true render — but zero ongoing per-fabric labor cost, which matters if the ~27–29-fabric hand-crafting effort turns out to be a harder sell than this report assumes.
- **First implementation step**: run SAM2 on `front_2button_notch.png` to check panel-boundary quality before committing to reworking the warp math.

### 5. Depth/normal-map refinement as an auxiliary pass (low priority)

- Not a fix for either core problem on its own — Marigold/Metric3D-class monocular depth already sits in the existing pipeline (`make_drape.py` + Marigold step) and is being used correctly. Nothing in this survey suggests a materially better tool for that specific sub-step exists yet; no action needed beyond what's already built.

### 6. IC-Light-style relighting as a final polish pass (optional, lowest priority)

- Only relevant if baked hero images (Recommendation 1/2) show any lighting inconsistency against the rest of the configurator's staged background — a nice-to-have, not a driver of the drape/wrap fix.

## 5. What NOT to do

- **UV-space/3D-mesh texture methods** (FabricDiffusion, GarmentPainter, TexGarment) — every one of these needs a 3D garment mesh + UV unwrap as input, confirmed by directly re-reading the FabricDiffusion abstract this session. This is precisely the category of tooling the business has permanently ruled out (no CLO3D, no Marvelous Designer, no 3D artist). Don't revisit unless that constraint changes.
- **ControlNet/T2I-Adapter/IP-Adapter as a *primary* fix for per-panel wrap** — no published pipeline encodes per-panel grain direction as a conditioning channel; building one would be a research project (inventing and training a new conditioning signal), not an integration. Fine as an auxiliary/relighting pass, wrong tool for the core problem.
- **Chasing true real-time WebGL 3D** (matching Zakeke/Emersya-style marketing, or Threekit's 3D Player) — even where a live implementation genuinely is real-time 3D, standing one up requires the exact 3D-asset pipeline (mesh, PBR authoring, a 3D artist or equivalent tooling) this business has already decided is out of reach. Threekit's own documented best practice — 3D as source, 2D as delivery — is itself an argument *for* AI-baking the fabrics that need it, not for building real-time 3D.
- **Copying Suitsupply's literal layer-count scale** (~25–38 layers × per-fabric folders, implying tens of thousands of baked assets across a large catalog) — wrong lesson. GCC's actual advantage is a *small, bounded* combination space, which is exactly why AI-baking is affordable *for the subset that needs it* here in a way it wouldn't be at Suitsupply's SKU count — it is not, on its own, a reason to bake the whole catalog.
- **AO/crevice darkening in folds, fold-scale pattern warp wiggle, per-pixel-luminance sheen tint, 4-way mirror tiles, guessed pattern-scale densities, lossy normal-map codecs, anchoring drape median to 128** — all already tried and rejected in this project with measured evidence (HANDOVER §6 / pattern-render-playbook memory); don't re-explore.
- **Multi-view/video consistency tooling** (SV3D, Wonder3D, Any2AnyTryon) for animated or multi-angle configurator views — pilot-stage, no garment-specific validation found anywhere in this survey. Not relevant to the drape/wrap problem at all; park until front-view drape/wrap is solved and animation is separately requested.
- **Treating Zakeke/Emersya's "real-time WebGL, no baked assets" as settled fact** — that claim in this report is vendor-documentation-sourced only, not confirmed against a live client implementation's actual network payload the way Suitsupply/Hockerty/Proper Cloth/Black Lapel/Oliver Wicks were. Don't cite it as harder evidence than it is if this report is reused later.
- **Treating "the AI renders the whole garment, so wrap is solved" as a universal, free result** — this was this report's own original overclaim (§7). It is true for solids/textures and false, expensive, and sometimes-still-unsatisfying for exactly the bold directional patterns the business is actually struggling with. Don't cite the "for free" version of this claim if this report is reused later — cite the split verdict in §3/§4 instead.
- **Baking every fabric on visual-quality grounds alone** — per the revised cost model in §4, most of that spend (the ~20–35 hours of cross-cut colour QA on the easy majority) buys a result the compositor already delivers at parity. Only bake the fabrics where the compositor is actually measured to fail.

## 7. Revision after adversarial review

This report's original version (§1 bottom line, §3 matrix, §4 recommendations) was reviewed adversarially against its own cited sources and found to have one load-bearing error and three material gaps. The competitor teardown (§2) held up and was left unchanged. Changes made:

1. **Corrected the load-bearing claim that AI-bake fixes per-panel wrap "for free."** The original §3 matrix and Recommendation 1 asserted this was "proven repeatedly in the catalog work: windowpanes/houndstooth/pinstripes correctly follow garment structure." Re-reading the `gcc-fabric-ai-product-images` memory this claim cited shows the opposite for exactly those fabrics: a documented 15-fabric Elite "bold-check/windowpane REWORK LIST," ~12–14 more patterned Prestige fabrics, DBS139A shipping with the user "unhappy," the first DBQ791A attempt called "TERRIBLE," and a fix that required ruler+SpyderCheckr photography plus a two-engine (`gpt-5.5-pro` + `gpt-image-2`) iterative pipeline per fabric. §3's matrix row and Recommendation 1/2 in §4 now carry a **split verdict**: solids/textures — yes, essentially free; bold directional patterns — no, expensive and sometimes still unsatisfying. Added a **required validation gate** (run the real recipe on a chalk-stripe and a windowpane onto `renders/front_2button_notch.png`, compare against the compositor) that must pass before any bake-based recommendation is trusted, since the catalog's recipe was proven on a walking pose, never on the configurator's standing one.

2. **Rebuilt the cost estimate to include human hours, not just API calls.** The original "$40–125 total, a rounding error" counted API cost only. The revised §4 cost model separates the cheap solids/textures (API-only, ~$34–104, but with a newly-added ~20–35 hours of cross-cut colour-QA burden, extrapolated from the catalog's own 31%/49% front-back recolour-fix rates on just two views) from the expensive bold-pattern subset (~27–29 fabrics, 1–4 hours each of ruler photography/iteration/QA, ~30–115 hours total) — a realistic total of **roughly 50–150 engineering/human hours**, not a rounding error, for the bake-everything path.

3. **Made explicit the previously-unstated dependency and untested assumption underlying any bake-based recommendation.** Per HANDOVER, only the `notch` cut has the regenerated, consistent house model; the other four are pending the 14-render regeneration (`plan/MODEL_BIBLE.md`) — baking on top of that inconsistency would fix a cross-cut mismatch permanently into expensive "final" assets. Separately, the catalog's proven fabric-swap recipe was validated only on the catalog's *walking* base model, never on the configurator's *standing* one — an untested transfer, not a proven fact. Both are now stated as hard prerequisites, folded into the validation gate.

4. **Surfaced the buried strategic question and added the missing recommendation.** Added an explicit discussion, ahead of the ranked list, of what the configurator is *for* once fabric+options become baked stills (a guided picker + real order-capture flow over a matched still-image library, not a live-render engine — a legitimate but different value proposition, now named rather than assumed). Added **difficulty-routing** as a new, first-ranked recommendation: since the compositor already measures at Suitsupply parity for colour, scale, sharpness, moiré, specular, and density (per HANDOVER's own audit), and pattern-wrap is only visible where a directional pattern exists, the report now recommends keeping the compositor for the ~85% of the catalog where it already works and reserving the expensive, human-in-the-loop AI-bake for only the ~27–29 fabrics where wrap is actually broken. Bake-everything is retained as a ranked alternative (now Recommendation 2, evaluated on the corrected cost model above), not the primary path.

Sections §2 (competitor teardown) and the evidence-tier honesty around Zakeke/Emersya/Suitsupply were left as-is — they held up under review.
